# Project Plan: Doc Assistant — AI-Powered Doctor Visit Summarizer & Scheduler

---

## 1. Project Title

**Doc Assistant: Audio-to-Action Post-Visit AI for Patients**

---

## 2. Target User, Workflow, and Business Value

**Who the user is:**
A patient — specifically one who has just attended a medical appointment and needs to understand what happened, what they need to do next, and whether they need to schedule a follow-up. The target user does not need to be medically trained; the system is designed for anyone who finds clinical language confusing, forgets what was discussed, or struggles to act on verbal instructions given during a fast-paced appointment.

**What recurring task this improves:**
After every doctor visit, a patient must mentally (or physically) reconstruct what was said, identify action items (pick up a prescription, get a blood test, reduce sodium intake), and decide whether to book a follow-up. Today this is done from memory, from handwritten notes, or from a dense After Visit Summary (AVS) PDF generated by the EHR. All three are unreliable.

**Where the workflow begins and ends:**
- **Begins:** Patient presses record on the app at the start of their appointment
- **Ends:** Patient has a plain-language summary, a checked to-do list, and (if needed) a follow-up appointment booked directly via MyChart — all from within one interface

**Why this matters and to whom:**
Poor post-visit comprehension is a documented driver of medication non-adherence, missed follow-ups, and preventable readmissions. A 2021 JAMA study found that patients recall fewer than half of instructions given during an appointment. For patients managing chronic conditions (hypertension, diabetes, heart disease), missing one instruction can have serious downstream consequences. This workflow — summarizing and acting on a medical visit — recurs with every appointment and is directly addressable with a well-scoped AI system.

---

## 3. Problem Statement and GenAI Fit

**The exact task:**
Given an audio recording of a doctor's appointment, the system must produce a plain-language visit summary, extract structured action items (medications to pick up, tests to schedule, lifestyle changes), and identify any follow-up appointments required — then optionally book those appointments via the patient's MyChart account.

**What part benefits from language models:**
The core value comes from Claude's ability to read an unstructured medical dialogue and extract semantically meaningful, patient-facing content: what the diagnosis means in plain English, which medication instructions are actionable, and when a follow-up is clinically indicated. This requires understanding clinical context, reasoning about urgency, and generating natural language output — none of which is achievable with rule-based extraction or keyword matching.

**Why a simpler non-GenAI tool is not enough:**
A template-based or regex extraction tool cannot handle the variation in how doctors phrase diagnoses, medication instructions, or follow-up recommendations. An EHR-generated AVS exists but is written for billing and legal purposes, not patient comprehension. A simple transcription-only tool produces a wall of text with no structure, prioritization, or plain-language translation. The language model is the minimum necessary tool to bridge the gap between medical dialogue and patient-facing action.

## 4. Planned System Design and Baseline

**System Architecture:**

```
Audio (browser recording or file upload)
        ↓
  OpenAI Whisper API  [transcription]
        ↓
  Claude API (claude-sonnet)  [structured extraction]
        ↓
  JSON: { summary, medications, todos, follow_up_appointments, important_notes }
        ↓
  Frontend renders:
    - Plain-language summary
    - Interactive to-do checklist
    - Medication cards with pickup flags
    - Follow-up appointment cards
        ↓
  [If follow-up needed]
  Epic FHIR API (SMART on FHIR OAuth2)
        ↓
  Patient selects available slot → appointment booked in MyChart
```

**Backend:** FastAPI (Python), served locally on `localhost:8000`
**Frontend:** Single-page HTML/JS app, no build step required
**APIs:** OpenAI Whisper (transcription), Anthropic Claude (analysis), Epic FHIR R4 sandbox (scheduling)

---

**Course Concepts Integrated:**

**Concept 1 — Tool Use / Function Calling (Week 4)**
The system uses tool-style integration across three external services. Whisper is invoked as a transcription tool via a `POST /transcribe` endpoint. The Epic FHIR API is used as an action-execution tool: the backend calls `GET /Slot` to retrieve available appointment times and `POST /Appointment` to book. Claude's role is the reasoning layer that determines *which* tools to trigger (i.e., whether a follow-up appointment is needed, and of what specialty), while the backend executes the actual tool calls. This reflects the Week 4 pattern of an LLM producing structured outputs that drive downstream tool invocations.

**Concept 2 — Anatomy of an LLM Call: Structured Outputs (Week 2–3)**
Claude is prompted with a strict system instruction that requires it to return a JSON object with fixed schema fields (`summary`, `medications`, `todos`, `follow_up_appointments`, `important_notes`). The prompt uses explicit output constraints (return only valid JSON, no extra text), field-level instructions (plain language, no jargon), and conditional logic (only include fields relevant to the transcript). Temperature is kept low to reduce hallucination in clinical extraction. This directly applies Week 2–3 material on controlling LLM output format and content for downstream programmatic use.

---

**Baseline for Comparison:**
The baseline is the standard patient post-visit experience: reading the After Visit Summary (AVS) PDF generated by the EHR system. The AVS contains a clinical summary, medication list, and follow-up instructions — but written in clinical language, without prioritization, and with no integration into task management or scheduling. The comparison is: given the same visit, does the AI system produce action items that are more complete, more actionable, and better understood by the patient than those extracted from the AVS alone?

---

**The App — What the User Sees:**
The user opens a single web page in their browser. A large microphone button lets them record their appointment in real time; alternatively they can upload an audio file. After recording, they click "Analyze Appointment." A loading indicator shows transcription progress, then Claude analysis. The results page shows: a 3–5 sentence plain-language summary, a diagnosis/reason badge, an interactive to-do checklist (checkboxes the patient can tick off), medication cards with dosage and pickup flags, follow-up appointment cards with a "Schedule via MyChart" button, and important notes (warnings, lifestyle changes). Clicking the schedule button opens an Epic OAuth2 login popup; after authentication, a slot picker modal shows available times for the relevant specialty, and the patient can book with one click.

## 5. Evaluation Plan

**What success looks like:**
The system succeeds if: (a) the plain-language summary accurately reflects the clinical content of the visit without introducing errors, (b) all action items mentioned by the doctor are captured in the to-do list, (c) no hallucinated action items are added, and (d) follow-up appointments are correctly identified and bookable.

**What will be measured:**

| Dimension | Metric | Method |
|---|---|---|
| Summary accuracy | Factual correctness vs. transcript | Human rubric (1–5 scale) |
| Action item recall | % of doctor-mentioned actions captured | Human annotation of ground truth |
| Action item precision | % of extracted items that are real (no hallucinations) | Human annotation |
| Follow-up identification | Correct specialty + timeframe extracted | Human rubric |
| Plain-language quality | Readability score (Flesch-Kincaid) | Automated |
| Booking success | Slot search and appointment POST succeed | Integration test |
| Latency | End-to-end time from upload to results | Logged in backend |

**Test set:**
10–15 synthetic visit transcripts covering a range of visit types: hypertension follow-up, post-surgery check-in, diabetes management, new prescription, referral to specialist, normal annual physical (no action items). Transcripts will be generated from public clinical dialogue datasets (e.g., MTS-Dialog, ACI-Bench) and manually reviewed for realism. A matching AVS-style document will be created for each transcript to serve as the baseline artifact.

**Baseline comparison:**
For each test transcript, a human evaluator will read both the AI-generated summary+checklist and the equivalent manually written AVS. They will score both on: completeness of action items, clarity to a non-medical reader, and ease of scheduling follow-ups. The AI system must match or exceed the AVS on recall and clarity to be considered useful.

**Model-as-judge:**
For scaled evaluation, a second Claude call will be used as a judge: given the original transcript and the extracted JSON, it will score factual accuracy and completeness on a 1–5 rubric. Human spot-checks will validate alignment between model-as-judge scores and human scores on a 20% sample.

## 6. Example Inputs and Failure Cases

**Example Inputs:**

1. **Hypertension follow-up** — Doctor says blood pressure is still high, adjusts lisinopril dosage from 10mg to 20mg, asks patient to reduce sodium, return in 6 weeks. *Expected output: medication update, dietary to-do, 6-week cardiology follow-up.*

2. **Post-op check-in** — Surgeon clears the incision, says no restrictions on activity, no follow-up needed unless pain returns. *Expected output: minimal to-do, no follow-up appointment scheduled.*

3. **Diabetes management** — Doctor orders HbA1c blood test, refers patient to a dietitian, renews metformin prescription. *Expected output: lab test to-do, pickup flag on metformin, dietitian referral appointment.*

4. **New diagnosis** — Doctor diagnoses seasonal allergies, prescribes cetirizine OTC, no follow-up needed, warns about drowsiness. *Expected output: summary explains allergy diagnosis, medication instructions, important note about drowsiness.*

5. **Annual physical with no issues** — Doctor confirms patient is healthy, no medications, no follow-up. *Expected output: clean summary, empty to-do list, no appointment card.*

---

**Anticipated Failure Cases:**

1. **Crosstalk and background noise** — Medical offices are not quiet. If a nurse interrupts or two people speak simultaneously, Whisper may produce garbled text. Claude may then extract incomplete or incorrect action items from the corrupted transcript. *Mitigation: display the raw transcript so the patient can verify; add a "flag for review" warning if transcript confidence is low.*

2. **Ambiguous medication instructions** — A doctor might say "take it as needed" or "you can try the new dose" without specifying the drug name clearly. Claude may hallucinate the drug name or miss the instruction entirely. *Mitigation: low-temperature generation, prompt instruction to leave fields blank rather than guess, human review flag for any medication without explicit name.*

3. **Multi-doctor or multi-visit audio** — If the patient accidentally records a corridor conversation or a previous visit, the system will process irrelevant content as if it were clinical instruction. *Mitigation: prompt asks Claude to extract only clearly clinical instructions; add timestamp or session boundary UI.*

4. **Specialty not mappable to FHIR** — If Claude extracts "see a pain specialist" or "talk to a counselor," the system may not find a matching SNOMED code or available Epic slot. *Mitigation: graceful fallback — display the recommendation as a manual to-do with a note to call the clinic directly.*

## 7. Risks and Governance

**Where the system could fail:**
- Whisper transcription errors in noisy clinical environments
- Claude hallucinating medication names, dosages, or follow-up timelines not present in the transcript
- Epic FHIR slot search returning no results for certain specialties in the sandbox
- OAuth token expiry mid-session causing silent booking failures

**Where the system should NOT be trusted:**
- As a substitute for reading the official After Visit Summary or discharge instructions
- For any dosage or medication decision — the system is informational only
- When the transcript quality is poor (flagged to user)
- For emergency or urgent care instructions — the system is scoped to routine outpatient visits

**Controls and governance boundaries:**
- All extracted content displays the original transcript so the patient can cross-check
- A visible disclaimer: *"This summary is AI-generated. Always confirm instructions with your healthcare provider."*
- No clinical decision-making — the system extracts and presents; it does not recommend
- Appointment booking requires explicit patient confirmation (modal confirmation step before POST)
- No audio or transcript is stored server-side beyond the session — all processing is ephemeral
- API keys stored in `.env` file, never committed to the repository
- Synthetic/sandbox data only during development and evaluation

**Data and privacy concerns:**
- Audio recordings of medical visits are PHI under HIPAA. The current prototype processes data locally and does not persist recordings. A production version would require a BAA with OpenAI and Anthropic, and storage compliance.
- Epic FHIR sandbox uses synthetic patients only; no real patient data is accessed during development.

**Cost:**
- Whisper API: ~$0.006/minute of audio — negligible for testing
- Claude API: ~$0.003–0.015 per analysis call — manageable on free/low tiers
- Epic FHIR sandbox: free with developer registration
